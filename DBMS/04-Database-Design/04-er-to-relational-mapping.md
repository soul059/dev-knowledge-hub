## ğŸ“Š Summary Table

| ER Component | Mapping Rule |
|--------------|--------------|
| **Strong Entity** | Create table with all simple attributes |
| **Weak Entity** | Table with owner's PK + partial key as composite PK |
| **Simple Attribute** | Column in entity's table |
| **Composite Attribute** | Flatten into multiple columns |
| **Multi-valued Attribute** | Separate table with FK to entity |
| **Derived Attribute** | Don't store, or use view |
| **Key Attribute** | PRIMARY KEY constraint |

| Relationship Type | Mapping Approach |
|-------------------|------------------|
| **1:1** | FK in one table (preferably total participation side) with UNIQUE |
| **1:N** | FK in the "N" side table |
| **M:N** | Create junction table with both FKs as composite PK |
| **N-ary** | Create relationship table with all participating FKs |

| Generalization Option | Use When |
|----------------------|----------|
| **Single Table** | Few subclasses, few specific attributes, overlapping OK |
| **Table per Subclass** | Many specific attributes, overlapping common |
| **Table per Concrete** | Disjoint, rarely query superclass |

## â“ Quick Revision Questions

1. **How do you map a weak entity?**
   <details>
   <summary>Click for Answer</summary>
   Create a table containing the weak entity's attributes plus the owner entity's primary key. The composite of owner's PK and the partial key becomes the primary key. Add FK referencing owner entity.
   </details>

2. **Where do you place the foreign key in a 1:N relationship?**
   <details>
   <summary>Click for Answer</summary>
   The foreign key goes on the "N" (many) side. The entity that can have multiple relationships to the other gets the FK. Example: In Dept(1)-to-Emp(N), FK goes in Employee table.
   </details>

3. **How do you map an M:N relationship?**
   <details>
   <summary>Click for Answer</summary>
   Create a new junction/bridge table containing FKs to both entities. These FKs together form the composite primary key. Relationship attributes (like grade) go in this table.
   </details>

4. **How do you handle multi-valued attributes?**
   <details>
   <summary>Click for Answer</summary>
   Create a separate table containing the entity's PK and the multi-valued attribute. Together they form the primary key. Example: Employee_Phone(Emp_ID, Phone).
   </details>

5. **What are the three options for mapping generalization?**
   <details>
   <summary>Click for Answer</summary>
   (1) Single table: All entities in one table with type column, (2) Table per subclass: Superclass table + separate tables for each subclass with FK, (3) Table per concrete class: Only subclass tables, no superclass table.
   </details>

6. **How do you handle relationship attributes in 1:N vs M:N?**
   <details>
   <summary>Click for Answer</summary>
   In 1:N: Relationship attributes go to the "N" side table. In M:N: Relationship attributes go to the junction table. Example: Grade in enrollment goes to Enrollment junction table.
   </details>
