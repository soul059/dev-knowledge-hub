## 1. What is a Data Model?

A **Data Model** is a collection of concepts for describing:
- The **structure** of the data (data types, relationships)
- The **operations** that can be performed on data
- The **constraints** that must hold on the data

### Characteristics

| Feature | Description |
|---------|-------------|

| **Navigation** | Top-down, single path |

### Advantages & Disadvantages

| Advantages | Disadvantages |
|------------|---------------|
| Simple and fast for 1:N relationships | No M:N relationships |
| Clear parent-child structure | Data redundancy |
| Good for hierarchical data (org charts) | Complex queries difficult |
| Fast retrieval with known path | Insertion/deletion anomalies |
| Data integrity enforced | Inflexible structure |

### Properties of Relations

1. **No Duplicate Tuples**: Each row is unique
2. **Attribute Values are Atomic**: No multi-valued attributes
3. **No Ordering of Tuples**: Rows have no specific order
4. **No Ordering of Attributes**: Columns have no specific order
5. **Unique Attribute Names**: Each column has unique name

### Advantages & Disadvantages

| Advantages | Disadvantages |
|------------|---------------|
| Simple and intuitive | May be slower for complex relationships |
| Data independence | Joins can be expensive |
| Powerful query language (SQL) | Not ideal for hierarchical data |
| Flexible and easy to modify | Impedance mismatch with OOP |
| Strong theoretical foundation | Normalization overhead |
| ACID compliance | Not optimal for unstructured data |

### NoSQL vs Relational Comparison

| Feature | Relational (SQL) | NoSQL |
|---------|-----------------|-------|
| **Schema** | Fixed schema | Flexible/schema-less |
| **Scaling** | Vertical (scale up) | Horizontal (scale out) |
| **Transactions** | ACID | BASE (Eventually consistent) |
| **Joins** | Complex joins supported | Limited or no joins |
| **Data Structure** | Tables with rows/columns | Various (documents, key-value, etc.) |
| **Best For** | Structured data, transactions | Unstructured data, big data |
| **Examples** | MySQL, PostgreSQL, Oracle | MongoDB, Cassandra, Redis |

## â“ Quick Revision Questions

1. **What are the three main components of a data model?**
   <details>
   <summary>Click for Answer</summary>
   (1) Structure - How data is organized, (2) Operations - What manipulations can be performed, (3) Constraints - What rules must be followed.
   </details>

2. **Why is the hierarchical model not suitable for M:N relationships?**
   <details>
   <summary>Click for Answer</summary>
   The hierarchical model uses a tree structure where each child can have only ONE parent. M:N relationships require a child to have multiple parents, which violates the tree structure.
   </details>

3. **What is the difference between a superkey and a candidate key?**
   <details>
   <summary>Click for Answer</summary>
   A superkey is any set of attributes that uniquely identifies a tuple. A candidate key is a minimal superkey - it has no unnecessary attributes. Every candidate key is a superkey, but not every superkey is a candidate key.
   </details>

4. **Name two advantages of the relational model over hierarchical/network models.**
   <details>
   <summary>Click for Answer</summary>
   (1) Data independence - changes to physical storage don't affect logical structure, (2) Powerful declarative query language (SQL) - users specify WHAT they want, not HOW to get it.
   </details>

5. **What does BASE stand for in NoSQL context?**
   <details>
   <summary>Click for Answer</summary>
   Basically Available, Soft state, Eventually consistent. It's an alternative to ACID, allowing for greater scalability at the cost of immediate consistency.
   </details>

6. **When would you choose a graph database over a relational database?**
   <details>
   <summary>Click for Answer</summary>
   When the data has complex, interconnected relationships that require traversal - such as social networks, recommendation systems, or fraud detection where you need to analyze connections between entities.
   </details>
