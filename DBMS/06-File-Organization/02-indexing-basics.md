## â“ Quick Revision Questions

1. **Why must secondary indexes be dense?**
   <details>
   <summary>Click for Answer</summary>
   Secondary indexes are on non-ordering fields, so data isn't sorted by the index key. We can't use sparse index because we can't assume where a key value appears based on block position. Every record needs an explicit pointer.
   </details>

2. **What is an anchor record?**
   <details>
   <summary>Click for Answer</summary>
   An anchor record (or block anchor) is the first record in a data block. Sparse indexes store only anchor records because, in a sorted file, we can binary search to find the block containing any key, then scan within the block.
   </details>

3. **How does a clustering index differ from a primary index?**
   <details>
   <summary>Click for Answer</summary>
   Primary index is on a unique ordering key (one record per value). Clustering index is on a non-key field where multiple records can share the same value. Both require sorted data files, but clustering groups duplicates together physically.
   </details>

4. **Why do multi-level indexes reduce search time compared to single-level with binary search?**
   <details>
   <summary>Click for Answer</summary>
   Binary search divides by 2 each step (logâ‚‚ N blocks). Multi-level indexes divide by fan-out (often 100+) each step (logâ‚â‚€â‚€ N). Higher fan-out means fewer levels and fewer disk reads. Example: 10,000 blocks needs 14 binary search reads vs 2 multi-level reads.
   </details>

5. **What is index selectivity and why does it matter?**
   <details>
   <summary>Click for Answer</summary>
   Selectivity = distinct values / total rows. High selectivity (near 1.0) means the index effectively narrows search. Low selectivity (like 2 genders in 100K rows) means most queries still scan many records, making the index overhead not worth it.
   </details>

6. **What are the costs of maintaining too many indexes?**
   <details>
   <summary>Click for Answer</summary>
   Extra indexes add: (1) Space overhead - each index uses disk space, (2) Write overhead - every INSERT/DELETE/UPDATE must update all indexes, (3) Maintenance cost - index splits, reorganization. Balance read gains against write costs.
   </details>
