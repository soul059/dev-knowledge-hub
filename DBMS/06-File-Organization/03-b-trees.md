## â“ Quick Revision Questions

1. **Why are B-Trees preferred over binary trees for disk-based indexing?**
   <details>
   <summary>Click for Answer</summary>
   Binary trees have small nodes (1 key, 2 children), resulting in tall trees with many disk reads. B-Trees have large nodes matching disk block size (100+ keys), giving high fan-out and short trees (3-4 levels for millions of records). Each node read is one I/O, so fewer levels = fewer disk reads.
   </details>

2. **What is the minimum fill requirement in a B-Tree of order m?**
   <details>
   <summary>Click for Answer</summary>
   Non-root nodes must have at least âŒˆm/2âŒ‰ children (and âŒˆm/2âŒ‰-1 keys). For order 5, minimum is 3 children (2 keys). The root must have at least 2 children if not a leaf. This ensures nodes are at least 50% full, maintaining efficiency.
   </details>

3. **Why does a B+-Tree have higher fan-out than a B-Tree?**
   <details>
   <summary>Click for Answer</summary>
   B+-Tree internal nodes contain only keys and child pointers (no data pointers). B-Tree internal nodes include data pointers for each key. Without data pointers, more keys fit per node. Example: 18-byte entries vs 26-byte entries means ~40% more keys per block.
   </details>

4. **How do leaf links in B+-Trees help with range queries?**
   <details>
   <summary>Click for Answer</summary>
   After finding the start of a range via tree traversal, we simply follow "next" pointers between leaves to collect all records in order. No need to traverse back up the tree. Cost is just O(log n) to find start plus sequential reading of result leaves.
   </details>

5. **What happens when a B+-Tree leaf node splits?**
   <details>
   <summary>Click for Answer</summary>
   The leaf splits into two nodes, with approximately half the keys each. The middle key is COPIED up to the parent (not moved - it stays in the right leaf too). If the parent overflows, it splits with the middle key MOVED up. This difference ensures all data keys remain in leaves.
   </details>
