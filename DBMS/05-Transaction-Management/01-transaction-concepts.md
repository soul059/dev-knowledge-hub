## â“ Quick Revision Questions

1. **What is a transaction?**
   <details>
   <summary>Click for Answer</summary>
   A transaction is a sequence of database operations that together form a single logical unit of work. It must either complete entirely (commit) or have no effect at all (rollback).
   </details>

2. **Explain the four ACID properties.**
   <details>
   <summary>Click for Answer</summary>
   Atomicity: All-or-nothing execution. Consistency: Database goes from valid state to valid state. Isolation: Transactions don't interfere with each other. Durability: Committed changes survive failures.
   </details>

3. **What is the difference between Committed and Partially Committed states?**
   <details>
   <summary>Click for Answer</summary>
   Partially Committed means all operations executed but changes may still be in memory. Committed means changes are written to permanent storage. System crash after Partially Committed loses changes; after Committed, changes survive.
   </details>

4. **When would you use a SAVEPOINT?**
   <details>
   <summary>Click for Answer</summary>
   When you want partial rollback capability within a transaction. It allows rolling back to a specific point without undoing earlier successful operations. Useful for complex transactions with multiple logical sections.
   </details>

5. **What happens if system crashes after Write but before Commit?**
   <details>
   <summary>Click for Answer</summary>
   The transaction is considered failed. Recovery system will undo (rollback) all changes made by the transaction, restoring the database to its state before the transaction started. This ensures Atomicity.
   </details>

6. **What is auto-commit mode?**
   <details>
   <summary>Click for Answer</summary>
   Auto-commit automatically commits each SQL statement as a separate transaction. Each statement is atomic on its own, but there's no way to group multiple statements into a single atomic unit without explicitly starting a transaction.
   </details>
