## â“ Quick Revision Questions

1. **Q:** What are the four fields in the UDP header?
   <details>
   <summary>Answer</summary>
   The UDP header has exactly **4 fields** (8 bytes total):
   
   1. **Source Port** (16 bits): Sender's port number
   2. **Destination Port** (16 bits): Receiver's port number
   3. **Length** (16 bits): Total datagram length (header + data)
   4. **Checksum** (16 bits): Error detection
   
   That's it! This simplicity is what makes UDP so fast and efficient.
   </details>

2. **Q:** Why would you choose UDP over TCP?
   <details>
   <summary>Answer</summary>
   Choose UDP when:
   
   1. **Speed is critical**: No connection setup delay
   2. **Some loss is acceptable**: Streaming, gaming
   3. **Real-time**: VoIP, video calls (old data is useless)
   4. **Simple request-response**: DNS, DHCP queries
   5. **Broadcast/multicast needed**: Network discovery
   6. **Application handles reliability**: QUIC, custom protocols
   7. **Low overhead**: Many small messages
   
   The key trade-off: Speed vs reliability
   </details>

3. **Q:** How does UDP preserve message boundaries while TCP doesn't?
   <details>
   <summary>Answer</summary>
   **UDP (Datagram-oriented)**:
   - Each send() creates one datagram
   - Receiver gets complete message per recv()
   - No merging or splitting of messages
   
   **TCP (Stream-oriented)**:
   - Data is a continuous stream of bytes
   - Send boundaries are not preserved
   - Receiver might get partial/combined data
   
   Example: Sending "Hello" then "World"
   - UDP: recv() = "Hello", recv() = "World"
   - TCP: recv() might = "HelloWor", recv() = "ld"
   </details>

4. **Q:** Why does UDP checksum include a "pseudo-header"?
   <details>
   <summary>Answer</summary>
   The pseudo-header includes:
   - Source IP address
   - Destination IP address
   - Protocol number (17 for UDP)
   - UDP length
   
   **Purpose**: To detect packets delivered to the wrong host!
   
   Without checking IP addresses:
   - Datagram could be corrupted in transit
   - IP destination might change
   - UDP would accept data meant for another host
   
   The pseudo-header ensures end-to-end verification.
   </details>

5. **Q:** Can UDP source port be zero?
   <details>
   <summary>Answer</summary>
   **Yes!** The UDP source port is optional.
   
   - If set to 0, it means "no reply expected"
   - Used for one-way notifications
   - Server won't know where to send response
   
   In contrast, the **destination port is always required** because the receiver needs it for demultiplexing.
   </details>

6. **Q:** What happens when a UDP datagram is lost or arrives corrupted?
   <details>
   <summary>Answer</summary>
   **UDP itself does nothing!**
   
   - **Lost datagram**: Silently dropped. No retransmission.
   - **Corrupted datagram**: Checksum fails â†’ silently discarded.
   
   **No notification to sender or receiver!**
   
   If reliability is needed, the **application** must handle it:
   - Implement acknowledgments
   - Set timeouts
   - Retry logic
   - Sequence numbers for ordering
   
   This is why it's called "unreliable" - UDP provides best-effort delivery only.
   </details>
