## â“ Quick Revision Questions

1. **Q:** What is the difference between flow control and congestion control?
   <details>
   <summary>Answer</summary>
   **Flow Control**:
   - Prevents sender from overwhelming the **receiver**
   - Based on receiver's buffer space (rwnd)
   - Receiver advertises window in ACK
   - "Slow down, I can't process that fast!"
   
   **Congestion Control**:
   - Prevents senders from overwhelming the **network**
   - Based on detected congestion (cwnd)
   - Sender infers congestion from loss/delay
   - "Slow down, the network is busy!"
   
   Both limits apply: Effective window = min(rwnd, cwnd)
   </details>

2. **Q:** Explain Slow Start and when it exits.
   <details>
   <summary>Answer</summary>
   **Slow Start**:
   - Initial cwnd = 1 MSS (or 10 in modern TCP)
   - For each ACK: cwnd = cwnd + 1 MSS
   - Effect: cwnd **doubles every RTT** (exponential growth)
   
   **Exit conditions**:
   1. cwnd â‰¥ ssthresh â†’ Enter Congestion Avoidance
   2. Timeout â†’ cwnd = 1, ssthresh = cwnd/2, restart Slow Start
   3. 3 duplicate ACKs â†’ Enter Fast Recovery
   
   Despite the name, Slow Start grows **fast** (exponentially)!
   </details>

3. **Q:** What is AIMD and why is it fair?
   <details>
   <summary>Answer</summary>
   **AIMD** = Additive Increase, Multiplicative Decrease
   
   **Additive Increase**: cwnd += 1 MSS per RTT (linear growth)
   **Multiplicative Decrease**: cwnd /= 2 on loss (halve)
   
   **Why it's fair**:
   - All senders increase at same rate (additive)
   - All senders decrease proportionally (multiplicative)
   - Over time, all flows converge to equal share
   - Creates stable equilibrium
   
   This creates the characteristic "sawtooth" pattern.
   </details>

4. **Q:** How does Fast Retransmit work?
   <details>
   <summary>Answer</summary>
   **Problem**: Timeout is slow (often seconds)
   
   **Solution**: Use duplicate ACKs as early loss signal
   
   1. Packet is lost
   2. Subsequent packets arrive out of order
   3. Receiver sends duplicate ACKs (asking for missing byte)
   4. After **3 duplicate ACKs**: assume loss
   5. Immediately retransmit missing segment
   6. Don't wait for timeout!
   
   Why 3? Fewer might be just reordering. 3 is high probability of actual loss.
   </details>

5. **Q:** What happens after Fast Retransmit in TCP Reno?
   <details>
   <summary>Answer</summary>
   **Fast Recovery** occurs:
   
   1. Set ssthresh = cwnd / 2
   2. Set cwnd = ssthresh + 3 MSS
   3. Retransmit lost segment
   4. For each additional dup ACK: cwnd += 1 MSS
   5. When new ACK arrives: cwnd = ssthresh
   6. Exit to Congestion Avoidance
   
   **Key difference from timeout**:
   - Timeout: cwnd = 1, go to Slow Start
   - Fast Recovery: cwnd halved, stay in Congestion Avoidance
   
   Fast Recovery is less severe (loss wasn't catastrophic)
   </details>

6. **Q:** What is the zero window condition and how is it handled?
   <details>
   <summary>Answer</summary>
   **Zero Window**: Receiver advertises rwnd = 0 (buffer full)
   
   Sender must stop sending until window opens.
   
   **Problem**: Window update ACK might get lost!
   
   **Solution**: Zero Window Probe
   - Sender periodically sends 1-byte probe segment
   - Forces receiver to send current window value
   - Prevents deadlock where both sides wait forever
   
   Probe timer typically starts at RTO, backs off exponentially.
   </details>
