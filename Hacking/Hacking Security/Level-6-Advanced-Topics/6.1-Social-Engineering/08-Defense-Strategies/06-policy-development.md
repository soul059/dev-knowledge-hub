# Unit 8: Defense Strategies — Topic 6: Policy Development

## Overview

**Security policy development** establishes the formal rules, procedures, and guidelines that govern how an organization protects itself against social engineering attacks. Policies provide the foundation for technical controls, training programs, and incident response procedures. Without clear policies, security efforts are inconsistent, unenforceable, and often ineffective.

---

## 1. Essential Security Policies

```
POLICIES FOR SOCIAL ENGINEERING DEFENSE:

ACCEPTABLE USE POLICY (AUP):
  → Defines acceptable use of company resources
  → Email, internet, phone, devices
  → Social media usage guidelines
  → Personal device usage (BYOD)
  → Consequences for violations

EMAIL SECURITY POLICY:
  → Handling external emails
  → Attachment and link guidelines
  → Reporting suspicious emails
  → Personal email usage on work devices
  → Email encryption requirements
  → Auto-forwarding restrictions

PASSWORD AND AUTHENTICATION POLICY:
  → Minimum password requirements
  → MFA enforcement
  → Password sharing prohibition
  → Password manager usage
  → Account lockout procedures
  → Credential sharing prohibitions

PHYSICAL SECURITY POLICY:
  → Badge/access card requirements
  → Visitor management procedures
  → Tailgating prohibition
  → Clean desk policy
  → Device lock requirements
  → Photography/recording restrictions

INFORMATION CLASSIFICATION POLICY:
  → Data classification levels
  → Handling requirements per level
  → Sharing and distribution rules
  → Storage and encryption requirements
  → Disposal/destruction requirements

SOCIAL MEDIA POLICY:
  → What can be shared publicly
  → Company information restrictions
  → Personal vs professional accounts
  → Risks of oversharing
  → Incident reporting for social media

INCIDENT REPORTING POLICY:
  → What constitutes a reportable incident
  → How to report (procedures/tools)
  → Mandatory reporting requirements
  → No-retaliation clause
  → Response expectations and SLAs
```

## 2. Policy Development Process

```
POLICY LIFECYCLE:

STEP 1: IDENTIFY NEED
  → Risk assessment findings
  → Incident response lessons
  → Regulatory requirements
  → Industry best practices
  → Audit findings
  → New threats or attack vectors

STEP 2: RESEARCH AND DRAFT
  → Review industry frameworks (NIST, ISO)
  → Benchmark against peer organizations
  → Consult stakeholders (IT, HR, Legal, Business)
  → Draft policy with clear language
  → Avoid jargon where possible
  → Include examples for clarity

STEP 3: REVIEW AND APPROVAL
  → Legal review (compliance, enforceability)
  → HR review (employment law, fairness)
  → IT review (technical feasibility)
  → Business review (operational impact)
  → Executive approval
  → Union/labor consultation (if applicable)

STEP 4: COMMUNICATION AND TRAINING
  → Distribute to all affected employees
  → Conduct training on new policy
  → Obtain acknowledgment (signature/digital)
  → Post on intranet/policy portal
  → Include in new employee onboarding
  → Create quick reference guides

STEP 5: ENFORCEMENT
  → Consistent application
  → Progressive discipline process
  → Documentation of violations
  → Fair and equitable enforcement
  → Regular compliance monitoring

STEP 6: REVIEW AND UPDATE
  → Annual policy review (minimum)
  → Update after significant incidents
  → Update for new threats/technology
  → Incorporate feedback from employees
  → Track policy effectiveness
```

## 3. Policy Structure Template

```
POLICY DOCUMENT TEMPLATE:

1. TITLE AND METADATA
   → Policy name
   → Version number and date
   → Owner/responsible party
   → Approval authority
   → Next review date
   → Classification level

2. PURPOSE
   → Why does this policy exist?
   → What risk does it address?
   → 2-3 clear sentences

3. SCOPE
   → Who does this apply to?
   → What systems/locations are covered?
   → Any exclusions?

4. DEFINITIONS
   → Key terms defined
   → Acronyms explained
   → Avoid ambiguity

5. POLICY STATEMENTS
   → Clear, actionable requirements
   → Use "shall" (mandatory), "should" (recommended)
   → Numbered for reference
   → Examples where helpful

6. PROCEDURES
   → Step-by-step procedures
   → How to comply with policy
   → Who to contact for help
   → Reporting procedures

7. ROLES AND RESPONSIBILITIES
   → Who is responsible for what?
   → Management responsibilities
   → Employee responsibilities
   → IT/Security responsibilities

8. ENFORCEMENT
   → Consequences for violations
   → Progressive discipline
   → Escalation procedures
   → Exception process

9. RELATED POLICIES
   → Cross-references to related policies
   → Supporting standards and guidelines
   → Regulatory references

10. REVISION HISTORY
    → Version tracking
    → Change descriptions
    → Approval records
```

## 4. Wire Transfer and Financial Controls

```
WIRE TRANSFER POLICY (Anti-BEC):

REQUIREMENTS:
  → Dual authorization for all transfers > $X
  → Verbal confirmation via known phone number
  → Never process based on email alone
  → Vendor payment change requires verification
  → 24-hour waiting period for new payees
  → Separation of duties (requester ≠ approver)

VERIFICATION PROCEDURES:
  → Call back on KNOWN number (not from email)
  → Verify with the actual person (not assistant)
  → Document verification for each transfer
  → Use pre-arranged code words for urgency
  → Any bypass requires executive approval

VENDOR CHANGE MANAGEMENT:
  → Written request from vendor required
  → Verification call to vendor
  → 30-day validation period
  → Dual approval for banking changes
  → Regular vendor information audit
```

## 5. Policy Effectiveness

```
MEASURING POLICY EFFECTIVENESS:

METRICS:
  → Policy acknowledgment completion rate
  → Compliance audit findings
  → Incident rates related to policy areas
  → Policy violation trends
  → Employee awareness survey scores
  → Phishing simulation results

COMMON POLICY FAILURES:
  ✗ Too long and complex (nobody reads it)
  ✗ Too vague (open to interpretation)
  ✗ Not communicated effectively
  ✗ Not enforced consistently
  ✗ Not updated for new threats
  ✗ Conflicts with business processes
  ✗ No exception process defined
  ✗ Written by IT without business input

BEST PRACTICES:
  ✓ Keep policies concise and clear
  ✓ Use plain language
  ✓ Provide practical examples
  ✓ Align with business processes
  ✓ Enforce consistently and fairly
  ✓ Review and update annually
  ✓ Get executive sponsorship
  ✓ Include exception procedures
  ✓ Provide training and resources
  ✓ Measure and track compliance
```
